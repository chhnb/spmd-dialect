2. 分离 Overhead：用 CUDA 4 种策略对比

写了和 Taichi 完全对齐的 CUDA OSHER 实现 (1092 行)，用 4 种执行策略测同一个 kernel：

// 策略 1: Sync (Taichi 默认)
for (step = 0; step < N; step++) {
  shallow_water_step<<<grid,block>>>(...);
  cudaDeviceSynchronize();  // ← 等 GPU 做完
}

// 策略 2: CUDA Graph (零 launch 开销)
cudaStreamBeginCapture(stream);
for (step = 0; step < N; step++)
  shallow_water_step<<<..., stream>>>(...);
cudaStreamEndCapture(stream, &graph);
cudaGraphLaunch(graph);    // GPU 自己回放

// 策略 3: Persistent (1 次 launch)
persistent_kernel<<<coop_grid, block>>>(...) {
  for (step = 0; step < N; step++) {
    shallow_water_step_body();
    grid_sync();  // GPU 内部全局同步
  }
}

3. 怎么测的：控制变量实验（不依赖 profiler）

核心思路：同一个 kernel，只切换执行策略，差值就是该层的损失。全程只用 cudaEvent 计时。

4. Overhead 是什么

每步 Taichi/Warp 走完这条路径（F1 hydro-cal 64², 3060 实测）：

Python~3 μs → DSL runtime~5 μs → CUDA driver launch~30 μs → GPU compute~50 μs → cudaSync~50 μs → 返回 Python
GPU compute
~50 μs (26%)
cudaSync
~50 μs (26%)
CUDA launch
~30 μs (16%)
Python+DSL
~8 μs

Graph 消除了 launch + sync → 只剩 GPU compute (~50 μs)。
Sync 模式 = compute + launch + sync ≈ 190 μs → overhead = 140 μs = 74%

cudaDeviceSynchronize() 是最大开销。Taichi/Warp 每个 kernel 后都调它——
因为 Python 要拿到 GPU 结果才能执行下一行代码。
F2 每步 2 个 kernel → 2 次 launch + 2 次 sync = overhead 翻倍。

5. 我们做了什么：把循环和同步搬到 GPU 上

不是"去掉同步"——步间同步是正确性必需的。我们做的是把同步从 host round-trip 换成 GPU 片上 barrier。

Sync Loop（Taichi/Warp 默认）

// 循环在 Host (Python) 上
for step in range(500):
  launch kernel_1  → GPU 跑完
  cudaDeviceSynchronize()
    // GPU → PCIe → host → PCIe → GPU
    // ~70 μs round-trip (3060)
  launch kernel_2  → GPU 跑完
  cudaDeviceSynchronize()
    // 又一次 round-trip

// 每步 2× host-device 往返

Persistent Kernel（我们的方案）

// 循环在 GPU kernel 内部
__global__ void persistent_kernel(...) {
  cg::grid_group grid = cg::this_grid();
  for (int step = 0; step < 500; step++) {
    compute(my_cells);   // Phase 1
    grid.sync();
      // GPU 片上 barrier（所有 block）
      // ~几百 ns, 不离开 GPU
    transfer(my_cells);  // Phase 2
    grid.sync();
  }
}
// Host 只 launch 1 次，最后收结果

grid.sync() 的硬件机制与约束

cudaDeviceSynchronize()cg::grid_group::sync()
路径GPU → PCIe → host → driver → PCIe → GPUGPU 片上 atomics + spin-wait
延迟~70 μs (3060) / ~10 μs (B200)~几百 ns
前提无所有 block 必须同时驻留 SM 上
编译无特殊要求-rdc=true -lcudadevrt
Launch<<<grid,block>>>cudaLaunchCooperativeKernel()

为什么要全部驻留？ GPU 没有抢占。如果 block A 在等 block B 到达 barrier，但 block B 还在排队等 SM 空位 → 死锁。

约束公式：blocks ≤ blocks_per_SM(B, R) × num_SMs

3060, F1 OSHER (106 regs, B=256): 每 SM 最多 1 block → 上限 28 blocks → 128² = 64 blocks 超限，非法

参考文档：CUDA C++ Programming Guide — Cooperative Groups (CUDA 9.0+)
代码来源：hydro_cuda_osher.cu 第 418-530 行 (persistent_kernel)，第 956-1008 行 (bench_persistent)
底层实现：每个 block 到达 barrier 后 atomic increment 计数器，然后 spin-wait 直到计数器 == gridDim.x

CUDA GraphPersistent Kernel
循环在哪Host 上 replay N 次GPU kernel 内部 for loop
每步开销~2 μs (graph replay)~几百 ns (grid.sync)
Grid size 约束无限制blocks ≤ coop_limit
Register cache不行（每次新 launch，寄存器清空）可以（thread 跨步存活）
DMA overlap不行（打断 replay）可以（Compute ∥ Copy Engine）

动态控制流受限 (CUDA 12.x conditional nodes)完全灵活（就是普通 CUDA 代码）

4. 核心 Case：CG Solver — Graph 做不了，只有 Persistent 能做

Conjugate Gradient 每步 5 个 kernel + 2 次 host readback（读标量回 CPU）：

Taichi 写法（当前）

for step in range(1000):
  cg_matvec()        # kernel 1
  cg_dots()          # kernel 2 (reduction)
  alpha = dot_r[None] / dot_pAp[None]
  # ↑ HOST READBACK: GPU→CPU 读标量
  # 触发 cudaSync + cudaMemcpy
  cg_update(alpha)   # kernel 3
  cg_dot_rnew()      # kernel 4 (reduction)
  beta = dot_rnew[None] / dot_r[None]
  # ↑ 又一次 HOST READBACK
  cg_direction(beta) # kernel 5

# 每步: 5 launch + 5 sync + 2 readback
# = ~500 μs overhead on 3060

Persistent Fusion（我们的方案）

// 只 launch 1 次！
cg_persistent<<<coop_grid, 256>>>
  (N, x, r, p, Ap, ..., STEPS) {
  for (step = 0; step < STEPS; step++) {
    matvec(p → Ap);
    grid_sync();
    rr = device_reduce(r*r);
    pAp = device_reduce(p*Ap);
    grid_sync();
    alpha = rr / pAp; // GPU 上算!
    grid_sync();
    x += alpha*p; r -= alpha*Ap;
    grid_sync();
    rnew = device_reduce(r*r);
    beta = rnew / rr; // GPU 上算!
    grid_sync();
    p = r + beta*p;
    grid_sync();
  }
}

3060 实测

NSync (5 kern + readback)CUDA GraphPersistent Fusion加速

CUDA Graph 不能用（alpha/beta 要回 CPU）。PERKS 不能用（只处理单 kernel）。

正确性验证通过：max relative diff = 3.18×10⁻⁶。

9. 新发现：Persistent Kernel 可以零开销保存数据

GPU 的 Compute Engine 和 DMA Copy Engine 是独立硬件，可以并行工作：

// Persistent kernel 在 Compute Engine 上跑
persistent_kernel() {
  for step:
    compute phase1 → grid_sync → compute phase2 → grid_sync
    if (step % 100 == 0):
      write to save_buffer[step%2]     // GPU 端写到 double buffer
      __threadfence_system()           // 确保 host 可见
      *mapped_flag = step              // 通过 mapped memory 通知 host
}
// 同时！DMA Copy Engine 在后台传输
host_thread:
  poll mapped_flag → cudaMemcpyAsync(save_buffer → host, copy_stream)  // 不阻塞 compute!
方案μs/step额外开销
Persistent (不保存)5.85—
Persistent + DMA 保存 (每100步)6.58+12.5%
Sync loop + 阻塞保存76.69+1210%
PERKS (ICS'23) 和 AsyncTaichi 都没做过 compute-communication overlap。
CUDA Graph 保存数据必须打断回放；Persistent 可以边算边传。

11. 方法论：自动 Strategy Selection

分析 kernel 特征 → 自动选择最优策略

有 host readback? (CG 的 alpha/beta, 迭代求解器的收敛判断)
│
├── YES → grid ≤ cooperative limit?
│         ├── YES → ★ Persistent Fusion         CG 64²: 40.8x
│         │         (标量搬到 GPU 上算，多 kernel → 1)
│         └── NO  → Async sync elimination        ~2x
│
└── NO  → grid ≤ cooperative limit?
          ├── YES → 比较 Graph vs Persistent
          │         LULESH 64²: Persistent 27.0x
          │         LULESH 128²: Persistent 超限 → Graph 10.1x
          └── NO  → ★ CUDA Graph                  Heat2D 256²: 10.8x

3060 OSHER: T_fixed=32.3μs, T_per_wave=38.4μs | heat2d: T_fixed=0.19μs, T_per_wave=0.61μs
