2. 怎么发现的：用 CUDA 实验分离 compute 和 overhead

写了和 Taichi 完全对齐的 CUDA 实现（完整 OSHER solver, 1092 行），用 4 种执行策略对比：

4 种策略

Sync 每步 launch + sync（Taichi 默认）

Async 去掉逐步 sync

Graph 录制 launch 序列，GPU 回放

Persistent 单次 launch，kernel 内循环

关键原理

CUDA Graph ≈ 纯 GPU 计算时间（没有 launch/sync 开销）

所以：overhead = Sync − Graph

OSHER 完整 solver 实测 (3060, fp64, 106 registers)：

NCellsSyncAsyncGraphPersistentOH 占比
321024109.4 μs49.5 μs40.7 μs38.4 μs (2.85x)63%
644096157.9 μs68.8 μs—69.2 μs (2.28x)56%
12816384257.5 μs155.1 μs147.5 μs (1.75x)148.9 μs43%
即使用完整 OSHER Riemann solver (106 registers, fp64)，overhead 仍占 43-63%。
这不是只有 trivial stencil 才有的问题。

3. Overhead 是什么：6 层路径分解

每个 timestep，Taichi/Warp 走完这条完整路径：

Python loop~3 μs → DSL runtime~5 μs → CUDA driver~15 μs → GPU compute可变 → cudaSync~45 μs ! → 返回 Python

3060 实测分解 (hydro-cal)：

cudaSync
52.6 μs (62%)
cudaLaunch
24.2 μs (29%)
GPU compute
7.6 μs (9%)

91% 的时间是 overhead，GPU 只算了 9%

最大开销是 cudaDeviceSynchronize()：52.6 μs。
Taichi/Warp 在每个 kernel 后都调这个——因为 Python 需要拿到结果才能继续下一行。

5. Case 2：CG Solver — Graph 做不了的 Case

Conjugate Gradient：每步 5 个 kernel + 2 次 host readback

CG 每步执行流程

for step in range(1000):
    cg_matvec(A, p, Ap)              # kernel 1: Ap = A*p
    cg_dots(r, p, Ap, rr, pAp)      # kernel 2: rr = dot(r,r), pAp = dot(p,Ap)
    alpha = rr.read() / pAp.read()   # ← HOST READBACK! GPU→CPU 读标量
    cg_update(x, r, p, Ap, alpha)   # kernel 3: x += alpha*p
    cg_dot_rnew(r, rnew)            # kernel 4: rnew = dot(r,r)
    beta = rnew.read() / rr.read()   # ← HOST READBACK! 又要回 CPU
    cg_direction(r, p, beta)        # kernel 5: p = r + beta*p

为什么 CUDA Graph 做不了

Graph 录制的是固定的 kernel 序列。但 alpha 和 beta 每步不同，需要 CPU 计算后传给下一个 kernel。Graph 不能在回放中途回 CPU。

Persistent Fusion 怎么解决

persistent_kernel() {  // 只 launch 1 次！
  for step in range(1000):
    Ap = matvec(A, p)                    // phase 1
    grid_sync()
    rr = device_reduce(r*r)              // phase 2: GPU 上做 reduction
    pAp = device_reduce(p*Ap)
    grid_sync()
    alpha = rr / pAp                     // ← GPU 上算！不回 CPU！
    grid_sync()
    x += alpha*p; r -= alpha*Ap          // phase 3
    grid_sync()
    rnew = device_reduce(r*r)            // phase 4
    beta = rnew / rr                      // ← GPU 上算！
    grid_sync()
    p = r + beta*p                       // phase 5
    grid_sync()
}

3060 实测结果

NSync (5 kern + readback)GraphPersistent加速

128²486.0 μs❌ 不可用14.1 μs34.5x

6. Case 3：LULESH — Strategy Selection 的价值

Lagrangian hydro，每步 4 个 kernel，无 host readback（和 CG 不同）：

for step in range(1000):
    reset_forces()       # kernel 1
    calc_forces(p → f)   # kernel 2: 压力梯度 → 力
    update_nodes(f → v,x) # kernel 3: 力 → 加速度 → 速度 → 位置
    eos(x → p)           # kernel 4: 新体积 → 密度 → 压力 (状态方程)
    # 没有 host readback → Graph 和 Persistent 都可以！

7. 新发现：Compute + DMA 可以并行

GPU 内部有独立的硬件引擎：

GPU 硬件:
├── Compute Engine (30 SMs) ← kernel 用这个
├── Copy Engine 0 (DMA)    ← H2D 拷贝
└── Copy Engine 1 (DMA)    ← D2H 拷贝

它们可以同时工作！

Persistent kernel 可以边算边传

persistent_kernel:
  step N: compute → write buf[0]
  step N+1: compute → write buf[1]
    ↕ 同时 DMA 传 buf[0] → host
  step N+2: compute → write buf[0]
    ↕ 同时 DMA 传 buf[1] → host

3060 实测

方案μs/step额外开销
Persistent (不保存)5.85—
Persistent + async DMA 保存6.58+12.5% (几乎零开销)
Sync loop + 阻塞保存76.69+1210%
PERKS (ICS'23) 和 AsyncTaichi (2020) 都没有做过 compute-communication overlap。
这是 persistent kernel 相比 CUDA Graph 的独有优势：Graph 保存数据必须打断回放，
persistent 可以利用 DMA 引擎零开销保存。

9. 方法论：自动 Strategy Selection

输入: simulation kernel 特征

  有 host readback? (CG 的 alpha/beta)
  │
  ├── YES → grid ≤ cooperative limit?
  │         ├── YES → ★ Persistent Fusion (CG: 40x)
  │         │         把标量计算搬到 GPU 上，5 kernel → 1
  │         └── NO  → Async sync elimination (2x)
  │
  └── NO  → grid ≤ cooperative limit?
            ├── YES → 比较 Graph vs Persistent
            │         ├── 3060 LULESH 64²: Persistent 赢 (27x)
            │         └── 3060 LULESH 128²: Persistent 超限 → ★ Graph (10x)
            └── NO  → ★ CUDA Graph (Heat2D 256²: 10.8x)

Cost Model (MAPE 16%)

T_compute = T_fixed + T_per_wave × ceil(blocks / SMs)   // 从 Graph 数据拟合
T_sync    = T_compute + K × (OH_launch + OH_sync)         // K = kernels/step
T_graph   = T_compute + OH_graph_replay
T_persist = T_compute + K × OH_grid_sync

已实现工具

strategy_selector.py: 输入 kernel 参数 + GPU → 输出策略 + 预测时间

sim_optimizer.py: trace Taichi 循环 → 自动检测 host readback → 推荐策略
