问题：GPU 仿真的 Launch Overhead Wall

GPU 仿真 kernel 的大部分时间花在 launch overhead 上，而不是计算。

Overhead 来源

  • Host→Driver→GPU: 每次 kernel launch ~5-15μs
  • Python GIL: Taichi/Warp 额外 +30-40μs
  • 多 kernel/step: ADI 每步 4097 次 launch
  • 同步开销: cudaDeviceSynchronize ~5μs

关键发现

  • 小 kernel (3-20μs compute) 开销占 60-77%
  • DSL (Taichi) 比 CUDA best 慢 2-22×
  • CPU 干扰下 Taichi 劣化 2.5×
  • 但大 mesh 上 (20w cells) 开销仅 4%

Related Work (1): GPU Kernel Launch Overhead

工作来源核心思想与我们的关系
CUDA Graphs NVIDIA, CUDA 10+ 将 kernel launch 序列捕获成图，一次性 replay 消除逐次 host-side launch 开销。Graph Full Capture 将 N 步全部捕获，amortize launch cost 到 ~0μs/step。 我们的 Strategy S3。基线方法。
Cooperative Groups
+ Persistent Kernel		 NVIDIA, CUDA 9+ 单次 cudaLaunchCooperativeKernel，kernel 内部 grid_sync() 做步间同步。零 launch overhead，但要求所有 block 同时驻留 → grid 大小受限。 我们的 Strategy S4。13/21 案例 N/A → Grid-Stride 解决此限制。
Blink
arxiv 2604.07609, 2025		 学术论文 LLM 推理中用 GPU-resident scheduler kernel + device-side cudaGraphLaunch (fire-and-forget) 消除 CPU 参与。TTFT P99 降低 8.47×。 我们实验了 fire-and-forget / tail launch / conditional loop，不适用仿真（步骤间有数据依赖）。但启发了 CPU 干扰实验。
PyGraph
EuroSys 2024		 学术论文 自动将 Python GPU 程序转换为 CUDA Graph。分析 kernel 之间的依赖关系，自动 capture + replay。 关注同一问题（Python DSL launch overhead），但从编译器角度解决。我们从 benchmark 角度量化。
AsyncTaichi
Taichi 异步引擎		 Taichi Team Taichi 的 async backend：kernel launch 入队而不立即执行，batch 后一起发射。类似 CUDA Async 策略。 Taichi 的 overhead 仍然很高（47μs vs CUDA 5μs），说明 Python→C++ bridge 是主要瓶颈，不只是 launch 本身。

Related Work (2): Stencil 优化与 Benchmark Suites

Stencil 优化技术

工作核心思想我们的使用
PERKS
Zhang et al., ICS'23		 Persistent kernel + 寄存器缓存: 把 stencil halo 数据放在寄存器中而非 shared memory，减少 bank conflict。对 Jacobi 2D/3D 有效。 C1/C2 有 PERKS baseline/gen/genwr 数据，作为 persistent 的学术对照。
EBISU
Matsumura et al., ICS'23		 Deep temporal blocking: 在一个 persistent kernel 中做多步 stencil 的时间维展开，减少全局 memory 访问。 C1/C2 有 EBISU 数据。与 PERKS 的区别: EBISU 做时间维融合，PERKS 做寄存器缓存。
Temporal Blocking
经典优化		 在 tile 内做多个 time step，减少 global memory 带宽需求。AN5D, Pochoir 等工具自动化此优化。 PERKS/EBISU 是其 GPU 持久化版本。
Kernel Fusion
编译器优化		 将多个小 kernel 合并为一个，减少 launch 次数和中间数据写回 global memory。 我们的 Persistent kernel 本质上是手动 kernel fusion。Grid-Stride 扩展了其适用范围。

Related Work (3): GPU 仿真 DSL 框架

框架来源编译路径特点我们的发现
Taichi
Hu et al., SIGGRAPH'19		 MIT → EPFL Python → AST → LLVM IR → PTX Python DSL, @ti.kernel 装饰器, 自动微分, sparse data structure. 编译质量高（LLVM 优化）, 但 Python dispatch 开销大。 小问题 10-22× 慢于 CUDA best。大问题 (207K cells) 追平 CUDA。开销来自 Python→C++ bridge。
Warp
NVIDIA, 2022+		 NVIDIA Python → C++ → nvrtc (JIT) NVIDIA 官方 Python 仿真框架. wp.launch() API. 支持 wp.struct 参数打包. Nvrtc 生成代码比 nvcc 多 ~4× PTX。 Struct packing 后 overhead 降低。Codegen 质量问题 (OSHER solver 4× PTX bloat)。7 个案例实现。
Triton
Tillet et al., OpenAI		 OpenAI Python → MLIR → LLVM → PTX Block-level programming model. 适合 GEMM/Attention 等 tile-based 操作. 不适合间接访问 (stencil on unstructured mesh)。 3 个案例 (C1/C8/C9)。HydroF1 比 CUDA 慢 80× (不适合间接寻址)。
TileLang
TVM-based, 2024+		 社区 Python → TVM TIR → PTX Tile-oriented DSL, T.Kernel + T.Parallel. TVM JIT 编译. Branchless via T.if_then_else. TVM JIT 限制: 复杂 Osher solver 编译超时 (>10min). C1 OK, C8/C9 N/A。
Kokkos
Edwards et al., JPDC'14		 Sandia/DOE C++ → native CUDA/HIP/SYCL C++ performance portability layer. parallel_for + fence(). DOE exascale apps (LAMMPS, LULESH) 使用。编译质量最高。 最接近 CUDA 的 DSL。HydroF1 207K: Kokkos 128μs vs CUDA 123μs (仅 4% overhead)。

核心贡献: Grid-Stride Persistent Kernel

解决 Persistent 的硬件限制：cooperative launch 要求所有 block 同时驻留，大 grid 时 N/A

之前: 固定 Grid (13/21 N/A)

// 每个 thread 固定处理一个 cell
int i = blockIdx.x * 16 + threadIdx.x;
int j = blockIdx.y * 16 + threadIdx.y;
v[i*N+j] = stencil(u, i, j);
grid.sync();

// N=2048 → 需要 16384 blocks
// A100 最多驻留 648 blocks → N/A!

现在: Grid-Stride Loop (永远可用)

// 每个 thread 处理多个 cell
int tid = blockIdx.x * 256 + threadIdx.x;
int stride = gridDim.x * 256;
for (int idx = tid; idx < N*N; idx += stride)
    v[idx] = stencil(u, idx);
grid.sync();

// 永远只用 432 blocks → 永远可用!

各框架实现细节

CUDA (5 策略, 21×2 sizes)

  • Sync/Async/Graph: 标准 launch pattern
  • Persistent: cudaLaunchCooperativeKernel + grid_sync()
  • Grid-Stride: cooperative launch + stride loop → 永远不 N/A
  • 编译: nvcc -O3 -arch=sm_80 -rdc=true -lcudadevrt
  • 每个 case 独立 .cu binary

Taichi (21 cases, GPU+CPU)

  • Python DSL → LLVM → PTX (JIT)
  • 每个 case 一个 *_taichi.py
  • ti.init(arch=ti.cuda) / ti.cpu
  • Overhead 来源: Python → C++ bridge + JIT compile
  • warmup 后重新初始化状态

Warp (7 cases, struct packing)

  • Python → C++ → nvrtc (JIT)
  • 用 wp.struct 打包参数 (减少 launch args)
  • 每个 case 一个 *_warp.py
  • 问题: nvrtc 生成代码比 nvcc 多 4× PTX
  • C16 双线性插值修复了 warp 7.83% 偏差

Triton / TileLang / PERKS / EBISU

  • Triton: C1/C8/C9, tile-based GPU programming
  • TileLang: C1 only (C8/C9 TVM JIT crash)
  • PERKS: 寄存器缓存 persistent stencil (C1/C2)
  • EBISU: temporal blocking stencil (C1/C2)
  • Kokkos: C++ portable parallelism (C1/C8/C9)

正确性验证: 21/21 Pass (100-step contract)

验证方法

  • 每个 case 至少 2 个独立实现
  • 100 步后比较输出 (max_rel_error < 5%)
  • Taichi GPU vs CPU vs Warp/Triton vs NumPy
  • 独立 NumPy 参考实现 (C11/C14/C18/C21)

Blink 启发: Device-Side Graph Launch 探索

受 Blink: CPU-Free LLM Inference (arxiv 2604.07609) 启发

尝试了什么

  • Fire-and-forget: ~2μs/step 但数据竞争 (仿真步骤间有依赖)
  • Tail Launch: 正确但 ~7μs/step (比 Host Graph 慢)
  • Conditional Graph Loop: ~5μs/step (仍然更慢)

为什么不适用?

Blink 的 decode steps 是独立的 (可以 fire-and-forget)
仿真步骤是顺序依赖的 (每步读上一步输出) → 必须串行 → GPU 端调度反而更慢

真正的收获

  • CPU 干扰实验: Taichi 在 CPU stress 下劣化 2.5×
  • Graph Full Capture 已是最优: 一次 host launch, GPU 内部执行所有步骤, ~0μs/step
  • 启发了 Grid-Stride: "如何消除硬件限制" 的思考方式

API 验证

cudaStreamGraphFireAndForget 和 cudaStreamGraphTailLaunch 在 A100 sm_80 + CUDA 12.6 上确认可用

cudaGraphConditionalHandle (while loop) 也已验证
