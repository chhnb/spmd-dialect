Overhead 从哪来？

Host → GPU 开销

  • 每次 kernel launch: CPU→Driver→GPU ~5-15μs
  • cudaDeviceSynchronize: ~5μs 等待
  • Python GIL: Taichi/Warp 额外 +30-40μs

多 Kernel 放大效应

  • Stable Fluids: 102 launches/step
  • ADI: 4097 launches/step
  • Gram-Schmidt: 6144 launches/step
  • 每次 launch 都付一遍 overhead

Persistent Kernel 的问题

cudaLaunchCooperativeKernel 要求所有 block 同时驻留

N=2048 Heat2D → 需要 16384 blocks

A100 最多同时驻留 648 blocks

→ N/A!

21 个案例中 13 个 在大 mesh 上无法使用 Persistent

解法：Grid-Stride Persistent Kernel

每个 thread 用 for 循环处理多个 cell，grid 大小固定

之前：固定 Grid

// 每个 thread 只处理一个 cell
int idx = blockIdx.x * 256 + threadIdx.x;
v[idx] = stencil(u, idx);

// N=2048 → 16384 blocks → N/A!

现在：Grid-Stride Loop

// 每个 thread 处理多个 cell
for (int idx = tid; idx < N*N; idx += stride)
    v[idx] = stencil(u, idx);

// 永远只用 432 blocks → 永远可用!

Blink 探索：Device-Side Graph Launch

我们尝试了 Blink 的 3 种 GPU 端调度方案:

方案Overhead正确性结论
Fire-and-forget~2μs✗ 数据竞争仿真步骤间有依赖，不能并行
Tail Launch~7μs✓比 Host Graph (~5μs) 还慢
Conditional Loop~5μs✓比 Host Graph 还慢

关键 insight: Blink 的 decode steps 是独立的（可以 fire-and-forget），
仿真步骤是顺序依赖的 → GPU 端调度反而更慢

关键修复

  • C4 Wave: 2-buffer → 3-buffer rotation（修复 race condition）
  • C14 PIC: fp32→fp64（lockstep CUDA+Taichi+NumPy）
  • C16 StableFluids: nearest→bilinear advection（修复 Warp 7.8% 偏差）
  • C21 GramSchmidt: rank-deficient init fixed + fp64（lockstep）
  • 11 个 Taichi 文件: warmup 后重新初始化 state
  • 每个修复都在 CUDA .cu + Taichi + NumPy 同步修改

策略选择决策树

问题规模大 & 单 kernel/step?
├── YES → Async (开销 <5%)
└── NO → grid ≤ maxCooperativeBlocks?
    ├── YES → Persistent (0μs, 最快)
    └── NO → Grid-Stride Persistent ★ (0μs, 无限制)

固定步数? → Graph Full Capture (~0μs)

总结

  • Benchmark: 21 案例 × 14 策略 × 多尺寸 = 319 数据点
  • 发现: Overhead wall 取决于 kernel 数 × compute 量，不是问题大小
  • Grid-Stride Persistent: 消除 13 个 N/A，比 Graph 快 1.1-2.3×
  • DSL: 小问题 Taichi 慢 10-22×，大问题追平 CUDA
  • Blink: Device-side 不适用仿真，但 CPU 干扰免疫有价值
  • 正确性: 21/21 pass，100-step contract，独立 NumPy 参考
