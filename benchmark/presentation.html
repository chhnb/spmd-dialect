问题：GPU 仿真的 Launch Overhead Wall

GPU 仿真 kernel 的大部分时间花在 launch overhead 上，而不是计算。

Overhead 来源

  • Host→Driver→GPU: 每次 kernel launch ~5-15μs
  • Python GIL: Taichi/Warp 额外 +30-40μs
  • 多 kernel/step: ADI 每步 4097 次 launch
  • 同步开销: cudaDeviceSynchronize ~5μs

关键发现

  • 小 kernel (3-20μs compute) 开销占 60-77%
  • DSL (Taichi) 比 CUDA best 慢 2-22×
  • CPU 干扰下 Taichi 劣化 2.5×
  • 但大 mesh 上 (20w cells) 开销仅 4%

核心贡献: Grid-Stride Persistent Kernel

解决 Persistent 的硬件限制：cooperative launch 要求所有 block 同时驻留，大 grid 时 N/A

之前: 固定 Grid (13/21 N/A)

// 每个 thread 固定处理一个 cell
int i = blockIdx.x * 16 + threadIdx.x;
int j = blockIdx.y * 16 + threadIdx.y;
v[i*N+j] = stencil(u, i, j);
grid.sync();

// N=2048 → 需要 16384 blocks
// A100 最多驻留 648 blocks → N/A!

现在: Grid-Stride Loop (永远可用)

// 每个 thread 处理多个 cell
int tid = blockIdx.x * 256 + threadIdx.x;
int stride = gridDim.x * 256;
for (int idx = tid; idx < N*N; idx += stride)
    v[idx] = stencil(u, idx);
grid.sync();

// 永远只用 432 blocks → 永远可用!

各框架实现细节

CUDA (5 策略, 21×2 sizes)

  • Sync/Async/Graph: 标准 launch pattern
  • Persistent: cudaLaunchCooperativeKernel + grid_sync()
  • Grid-Stride: cooperative launch + stride loop → 永远不 N/A
  • 编译: nvcc -O3 -arch=sm_80 -rdc=true -lcudadevrt
  • 每个 case 独立 .cu binary

Taichi (21 cases, GPU+CPU)

  • Python DSL → LLVM → PTX (JIT)
  • 每个 case 一个 *_taichi.py
  • ti.init(arch=ti.cuda) / ti.cpu
  • Overhead 来源: Python → C++ bridge + JIT compile
  • warmup 后重新初始化状态

Warp (7 cases, struct packing)

  • Python → C++ → nvrtc (JIT)
  • 用 wp.struct 打包参数 (减少 launch args)
  • 每个 case 一个 *_warp.py
  • 问题: nvrtc 生成代码比 nvcc 多 4× PTX
  • C16 双线性插值修复了 warp 7.83% 偏差

Triton / TileLang / PERKS / EBISU

  • Triton: C1/C8/C9, tile-based GPU programming
  • TileLang: C1 only (C8/C9 TVM JIT crash)
  • PERKS: 寄存器缓存 persistent stencil (C1/C2)
  • EBISU: temporal blocking stencil (C1/C2)
  • Kokkos: C++ portable parallelism (C1/C8/C9)

正确性验证: 21/21 Pass (100-step contract)

验证方法

  • 每个 case 至少 2 个独立实现
  • 100 步后比较输出 (max_rel_error < 5%)
  • Taichi GPU vs CPU vs Warp/Triton vs NumPy
  • 独立 NumPy 参考实现 (C11/C14/C18/C21)

关键修复

  • C4 Wave: 2-buffer → 3-buffer rotation (race condition)
  • C11 FDTD: amplitude 1.0 → 0.01 (stability, lockstep)
  • C14 PIC: fp32 → fp64 (atomic deposit, lockstep)
  • C16 StableFluids: nearest → bilinear advection
  • C21 GramSchmidt: rank-deficient init fixed, fp64 lockstep
  • 11 Taichi files: warmup 后重新初始化 state

Blink 启发: Device-Side Graph Launch 探索

受 Blink: CPU-Free LLM Inference (arxiv 2604.07609) 启发

尝试了什么

  • Fire-and-forget: ~2μs/step 但数据竞争 (仿真步骤间有依赖)
  • Tail Launch: 正确但 ~7μs/step (比 Host Graph 慢)
  • Conditional Graph Loop: ~5μs/step (仍然更慢)

为什么不适用?

Blink 的 decode steps 是独立的 (可以 fire-and-forget)
仿真步骤是顺序依赖的 (每步读上一步输出) → 必须串行 → GPU 端调度反而更慢

真正的收获

  • CPU 干扰实验: Taichi 在 CPU stress 下劣化 2.5×
  • Graph Full Capture 已是最优: 一次 host launch, GPU 内部执行所有步骤, ~0μs/step
  • 启发了 Grid-Stride: "如何消除硬件限制" 的思考方式

API 验证

cudaStreamGraphFireAndForget 和 cudaStreamGraphTailLaunch 在 A100 sm_80 + CUDA 12.6 上确认可用

cudaGraphConditionalHandle (while loop) 也已验证
